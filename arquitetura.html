Data sources (origens de dados)

Event producers — onde os dados nascem e são capturados em tempo real

0ms
origem
▶
⚙
O que é
Todas as fontes de dados que alimentam a plataforma. No Fast Data, a captura é feita via Change Data Capture (CDC) em bancos de dados, webhooks de sistemas, sensores IoT, logs de aplicações e eventos de plataformas. Os dados são capturados no momento em que são gerados, não em janelas de batch.
🔌
Fontes típicas
CDC em bancos: Oracle, PostgreSQL, MySQL, SQL Server (via Debezium, Golden Gate)
Sistemas: ERPs (SAP), CRMs (Salesforce), SaaS diversos
IoT: sensores, telemetria, dispositivos de campo
Logs: aplicações, infra, segurança
Storage: buckets S3, GCS, Azure Blob (eventos de mudança)
DebeziumGolden GateMQTTWebhooksRESTgRPC
🔒 Segurança nesta camada
Autenticação na origem (credentials vault), criptografia em trânsito (TLS/mTLS), controle de acesso por conector. No DHuO, conectores nativos com políticas de segurança pré-configuradas.
streaming contínuo
02

Ingestão de dados (streaming)

Apache Kafka — barramento de eventos distribuído para captura e distribuição em tempo real

~5ms
latência acumulada
▶
⚙
O que é
O Apache Kafka é o backbone de streaming do Fast Data. Ele recebe todos os eventos das fontes e os distribui para os consumidores downstream de forma ordenada, durável e escalável. Diferente de uma fila tradicional, o Kafka mantém um log imutável que permite event replay — recuperação de falhas sem perda de dados.
🛠
Características
Throughput: milhões de eventos/segundo por cluster
Latência: sub-millisecond para produção de eventos
Durabilidade: replicação multi-broker, persistência em disco
Event replay: reprocessamento de eventos históricos sob demanda
Conectores: Kafka Connect para integração com 100+ fontes/destinos
Apache KafkaKafka ConnectSchema RegistryConfluentAmazon MSK
🔒 Segurança nesta camada
SASL/SCRAM ou mTLS para autenticação de produtores/consumidores. ACLs por tópico. Criptografia em trânsito (TLS) e opcionalmente at-rest. Schema Registry garante integridade dos contratos de dados.
stream processing
03

Processamento e análise (stream processing)

Apache Flink / Spark Streaming — transformação, enriquecimento e analytics em tempo real

~15ms
latência acumulada
▶
⚙
O que é
Camada onde os dados brutos são transformados, enriquecidos, filtrados e analisados em tempo real, enquanto fluem. Diferente do batch (onde você processa lotes acumulados), aqui cada evento é processado individualmente assim que chega. É onde rodam regras de negócio, detecção de anomalias, agregações em janelas temporais e cálculos de métricas.
🛠
Capacidades
Stream analytics: agregações, joins e windowing em tempo real
CEP: Complex Event Processing (padrões em sequências de eventos)
Enriquecimento: lookup em dados de referência para contexto
Exatamente uma vez: exactly-once semantics (sem duplicatas)
Estado: processamento stateful com checkpointing automático
Apache FlinkSpark StreamingKSQLKafka Streams
🔒 Segurança nesta camada
Isolamento de jobs por namespace/tenant. Criptografia de estado em checkpoints. Controle de acesso a tópicos de entrada/saída. Mascaramento de dados sensíveis durante a transformação (PII/LGPD).
dados processados
04

Armazenamento rápido (serving layer)

Bancos in-memory e NoSQL — dados de contexto disponíveis em milissegundos para consumo

~25ms
latência acumulada
▶
⚙
O que é
Esta é a camada de contexto do Fast Data — o coração do conceito. Dados de curto prazo ficam aqui disponíveis para serviços, APIs, agentes de IA e aplicações. São dados quentes (hot data) que perdem valor com o tempo. Após o uso, podem ser persistidos no Data Lake ou banco legado para histórico. É armazenamento in-memory: leitura em sub-milissegundos.
🛠
Tecnologias
In-memory: Redis, Memcached, Hazelcast (sub-ms de leitura)
NoSQL: MongoDB, Cassandra, DynamoDB (ms de leitura)
TTL (Time to Live): dados expiram automaticamente após período configurável
Padrão: dados quentes ficam disponíveis por minutos/horas, depois fluem para warm storage
RedisMemcachedHazelcastMongoDBCassandraDynamoDB
🔒 Segurança nesta camada
Criptografia at-rest (AES-256). Network isolation (VPC/private subnet). Autenticação por token/password. Dados sensíveis com mascaramento ou tokenização. TTL automático previne retenção indevida.

DHuO — API Gateway, iPaaS e Governança

Plataforma da Engineering para orquestração, exposição, segurança e governança de APIs, dados e AI

~40ms
latência acumulada
▶
⚙
O que é
O DHuO é a plataforma proprietária da Engineering para integração de sistemas, dados e AI. No Fast Data, ele atua como a camada de exposição e governança: todas as APIs que servem dados em tempo real passam pelo DHuO, com autenticação, rate limiting, catálogo, observabilidade e audit trail. Também orquestra integrações via conectores nativos, MCP Server e protocolo A2A para agentes.
🛠
Capacidades
API Management: design, gateway, catálogo, developer portal
iPaaS: orquestração no-code, conectores, multi-cloud
AI Gateway: MCP Server, AI API Design, Agent Builder
Data Integration: pipelines, data quality, data catalog
Protocolos: REST, GraphQL, WebSocket, Webhooks, gRPC, events

06

Consumo e ação (output real time)

BI, Machine Learning, Agentes de IA, ações automáticas — onde o dado vira valor de negócio

<80ms
end-to-end
▶
⚙
O que é
A camada onde dados em tempo real se transformam em ação imediata. Dashboards operacionais atualizados a cada segundo. Modelos de ML fazendo inferência sobre dados do presente. Agentes de IA com contexto completo e atualizado. Sistemas automatizados disparando ações sem intervenção humana. O dado sai da origem e chega aqui em menos de 80ms.
🛠
Consumidores
BI real time: dashboards operacionais (dataviz streaming)
Machine Learning: modelos com feature store em tempo real
Deep Learning / Visão Computacional: inferência online
Agentes de IA: contexto atualizado via MCP + A2A
Ações automáticas: alertas, bloqueios, triggers, notificações
Apps e canais digitais: dados frescos em portais, apps, WhatsApp
🔒 Segurança nesta camada
Controle de acesso por consumidor/aplicação. Scopes e permissões por API. Audit trail de quem consumiu o quê e quando. Anonimização para BI público. Compliance LGPD/GDPR no output.
Ciclo de vida do dado
⚡ Hot — Fast Data
Dados de contexto de curto prazo. Disponíveis em milissegundos para serviços, AI e BI. Expiram via TTL após uso.
Redis, Hazelcast, Kafka | <100ms | minutos a horas
→
🗃 Warm — Data Lake / DW
Dados persistidos para análise histórica, consolidação e treinamento de modelos. Incremental, ~1min a 24h+.
S3, BigQuery, Snowflake, Databricks | min–horas | meses
→
🗃 Cold — Legado / Arquivo
Dados transacionais, compliance e arquivo de longo prazo. Persistência definitiva em bancos legados ou cold storage.
Oracle, SQL Server, Glacier, Archive | horas–dias | anos
Padrões arquiteturais suportados
Event-Driven
Arquitetura orientada a eventos. Cada mudança gera um evento imutável no Kafka.
Kappa
Stream-only. Todo processamento via streaming, sem camada batch separada.
Lambda
Batch + speed layer. Para cenários que exigem ambas as modalidades coexistindo.
Data Mesh
Federado por domínio. Cada área é responsável por seus data products em streaming.
